④ 배운 것 / 한계

환경 통제가 결과를 좌우한다 (가장 큰 교훈)

같은 코드라도 환경이 오염되면 점수·비용이 망가진다. 일상 Claude Code 설정(effort 기본값·보안 스킬·열린 WebSearch)이 코드 옵션을 덮어쓰거나 정책 필터를 예민하게 만들었다. 이를 통제하자 special\_rsa가 풀렸고(오염 환경에선 매번 포기) 비용은 문제당 절반으로 떨어졌다. → 측정의 전제는 "코드"가 아니라 "통제된 환경"이다.



부속 발견:



allowed\_tools는 도구를 제한하지 않는다 (SDK 사양: 자동승인 목록일 뿐). 도구를 막으려면 disallowed\_tools(블랙리스트)를 써야 한다.

effort는 adaptive thinking의 상한이지 강제가 아니다 — 쉬운 문제엔 xhigh여도 thinking이 적어 비용·결과 영향이 작다.

구독(OAuth) 인증은 비용 측정을 무효화한다 — 자동 측정은 API 키로.

모델 한계 (프롬프트로 못 고치는 것)

caesar·miniRSA는 프롬프트를 4번 바꿔도 동일하게 실패했다. 모델이 복호화 뒷부분을 환각으로 채우거나(caesar) 큰 수 연산 정밀도를 못 맞춘다(miniRSA). 프롬프트 튜닝의 한계 지점 — 다음 레버는 도구 사용 강제·검증 루프(아키텍처)지 프롬프트가 아니다.



채점의 함정

정답이 순서 무관인 문제(두 소수 등)는 모델이 다른 순서로 내면 오답으로 잡힌다 (john\_pollard). 채점 로직이 "맞은 것을 틀렸다"고 할 수 있다 — 불일치는 왜 다른지 확인해야 한다.



정책 오탐

서사형 한국어 description이 프롬프트 인젝션으로 오탐돼 비결정적 거부를 유발한다 (이세계). 사이버 검증과 무관. 다른 분야의 한국어 문제도 같은 리스크가 있다.



측정 노이즈

같은 조건에서도 실행마다 결과가 바뀌는 문제가 있다(AES-ABC: O/X 출렁임, 이세계: 거부/성공). 1회 측정으로 우열을 판단하면 안 된다 — 1점 차이는 대개 노이즈다. 정식 비교는 다회 반복 평균이 필요하다(시간 제약으로 이번엔 미실시).
