K!t$un3b1 - 자동 Forensics CTF 솔버 실험 결과

1\. 뭘 만들었나

이번 프로젝트에서는 CTF 문제를 자동으로 분석하고 풀이를 시도하는 자동 CTF 솔버를 만들었다.



솔버는 Claude Agent SDK와 Opus 4.7을 이용해 문제 파일을 읽고, 필요한 명령어나 Python 스크립트를 직접 실행하면서 플래그를 찾는 방식으로 동작한다. 모델이 플래그를 찾으면 FLAG: <value> 형식으로 출력하고, 이후 정답 파일과 자동으로 대조해 해결 여부를 판단한다.



이번 실험에서는 여러 CTF 분야 중 Forensics 분야를 대상으로 측정했다. Forensics 문제는 이미지, 로그, 디스크 이미지, 바이너리 파일, 압축 파일, 메타데이터, 숨겨진 데이터 등 다양한 형태로 출제되기 때문에, 단순히 하나의 도구만 실행하는 방식으로는 해결하기 어렵다.



따라서 솔버는 먼저 문제 설명과 문제 파일을 확인한 뒤, 파일 형식, 문자열, 메타데이터, 매직 바이트, 의심스러운 오프셋 등을 분석하고, 그 결과에 맞는 도구를 선택해 풀이를 진행하도록 구성했다.



문제 구조는 다음과 같이 구성했다.



forensic/

├── problem\_01/

│   ├── Dockerfile

│   ├── meta.json

│   └── 문제 파일

├── problem\_02/

│   ├── Dockerfile

│   ├── meta.json

│   └── 문제 파일

└── ...



answer/

├── problem\_01.txt

├── problem\_02.txt

└── ...

각 문제 폴더에는 문제 설명이 담긴 meta.json과 실제 분석할 문제 파일이 들어간다. 정답은 answer/ 폴더에 따로 저장해, 모델이 문제 풀이 중 정답 파일을 직접 참고하지 못하도록 분리했다.



2\. 어떻게 측정했나

측정은 Forensics CTF 문제 20개를 대상으로 진행했다.



각 문제에 대해 솔버를 실행한 뒤, 모델이 출력한 플래그와 answer/problem\_NN.txt에 저장된 실제 정답을 비교했다. 모델 출력에서 FLAG: <value> 형식의 값을 추출하고, 정답 파일과 정확히 일치하면 해결한 것으로 판단했다.



측정 기준은 다음과 같다.



기준	설명

해결률	전체 문제 중 몇 문제를 자동으로 해결했는지

비용	문제 풀이에 사용된 API 비용

step 수	문제 풀이 중 실행된 도구 호출 횟수

stop reason	플래그 출력, step 제한, loop 감지, 포기 등 종료 이유

결과는 다음 세 가지로 나눌 수 있다.



결과	의미

solved	플래그를 찾고 정답과 일치한 경우

partially solved	플래그는 찾지 못했지만 의미 있는 증거를 찾은 경우

unsolved	유의미한 증거를 찾지 못했거나 풀이가 막힌 경우

또한 실험의 공정성을 위해 WebSearch와 WebFetch는 막았다. 즉, 모델이 인터넷에서 write-up을 검색해서 푸는 것이 아니라, 주어진 문제 파일만 분석하도록 했다.



반복 실행으로 인한 비용 증가를 막기 위해 step 제한과 loop 감지도 적용했다. 같은 명령이나 같은 도구 호출을 반복하면 loop로 판단하고, 더 이상 같은 행동을 반복하지 않도록 했다.

4\. 배운 것 / 한계

이번 실험을 통해 자동 CTF 솔버에서 중요한 것은 단순히 많은 도구를 실행하는 것이 아니라, 현재 발견된 증거에 맞는 도구를 선택하는 것이라는 점을 알 수 있었다.



예를 들어 파일 내부에서 ZIP 시그니처가 발견되면 해당 오프셋 기준으로 카빙을 시도해야 하고, PNG 파일에서 IEND 이후 데이터가 남아 있다면 appended data를 확인해야 한다. PCAP 문제에서는 전체 문자열을 무작정 찾는 것보다 프로토콜, TCP stream, 전송 파일을 중심으로 확인하는 것이 더 효율적이다.



즉, 자동화의 핵심은 “도구를 많이 실행하는 것”이 아니라 “관찰된 증거와 다음 행동을 연결하는 것”이다. 이 연결이 잘 될수록 불필요한 반복을 줄이고, 풀이 비용과 시간을 줄일 수 있다.



또한 환경 통제의 중요성도 확인했다. 같은 코드와 같은 문제를 사용하더라도 실행 환경이 다르면 결과가 달라질 수 있다. 설치된 도구의 종류와 버전, 실행 경로, 권한, Claude 설정, WebSearch 허용 여부 등에 따라 해결률과 비용이 달라질 수 있기 때문이다.



따라서 실험 결과를 공정하게 비교하려면 Docker와 같은 고정된 실행 환경을 사용하고, API 키, Claude 설정 디렉터리, 사용 가능한 도구, step 제한, 비용 제한을 모두 통제해야 한다.



이번 실험의 한계는 다음과 같다.



한계	설명

환경 완전 격리 부족	Dockerfile은 있지만, 현재 실행 구조에서는 실제 Bash 실행 환경의 영향을 받을 수 있음

비용 편차	MZGZ, Enc-JPG처럼 일부 문제에서 비용이 크게 증가함

플래그 검증 문제	플래그처럼 보이는 문자열이 실제 정답인지 추가 검증이 필요함

수동 추론 한계	문제 의도 해석이나 복잡한 복구가 필요한 문제는 자동화가 어려움

1회 측정의 불안정성	모델 출력이 비결정적일 수 있어 여러 번 측정해야 더 정확함

앞으로는 Docker 기반 실행을 더 강하게 고정하고, Forensics 세부 분야별 프롬프트를 분리하며, 비용이 많이 드는 문제 유형에는 step 제한과 effort 조정을 적용할 필요가 있다.



결론적으로 이번 실험에서는 Forensics CTF 문제 20개 중 14개를 해결하여 70%의 해결률을 보였다. 이를 통해 자동 CTF 솔버가 단순한 파일 분석, 로그 분석, 메타데이터 분석 문제에서는 효과적으로 동작할 수 있음을 확인했다. 다만 복잡한 복구나 수동 추론이 필요한 문제에서는 아직 한계가 있으며, 환경 통제와 비용 관리가 중요한 개선 과제로 남았다.
